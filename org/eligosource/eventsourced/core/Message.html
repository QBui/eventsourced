org.eligosource.eventsourced.core

Message

case class Message(event: Any, sender: Option[ActorRef], senderMessageId: Option[String], sequenceNr: Long, processorId: Int, acks: Seq[Int], ack: Boolean) extends Product with Serializable

A message to communicate application events. Application events are not interpreted by the library and can have any type.

Messages sent to an Eventsourced processor are called input messages. Processors process input messages by deriving zero or more output messages from it that the processor emits to a channel. When a channel successfully delivered (or stored, in case of a reliable channel) the emitted output message(s), an acknowledgement (with a reference to the input message) is written to the journal. During a replay (recovery), output messages that are derived from already acknowledged input messages are ignored by channels, avoiding unnecessary re-deliveries to channel destinations.

Since the eventsourced library doesn't make any assumptions about the structure and semantics of event, applications may also choose to send commands with Messages. In other words, the library can be used for both, event-sourcing and command-sourcing.

event

Application event (or command).

sender

Optional, application-defined sender reference that can be used by event processors to send responses to initial event message senders.

senderMessageId

Optional, sender-defined message id that allows receivers to detect duplicates (which may occur during recovery or fail-over).

sequenceNr

Sequence number which is generated when messages are written to a journal. Can also be used for detecting duplicates, in special cases.

processorId

id of the event processor that stored (and emitted) this message The processor id is given by Eventsourced.id.

acks

List of channel ids that have acknowledged message delivery (or storage). This sequence is only non-empty during recovery (i.e. message replay). Usually not used by applications.

ack

Whether or not a channel should write an acknowledgement to the journal. Used by event processors to emit a series of messages (derived from a single input message) where only for the last emitted message an acknowledgement should be written.
